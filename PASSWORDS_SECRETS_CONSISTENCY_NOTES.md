# Password Secrets Consistency Notes

## Summary

The repository now uses a shared SOPS dotenv helper for normal password-file read and write operations.

Shared helper:

- `scripts/lib/secrets-dotenv.sh`

Centralised operations:

- validate required SOPS paths
- decrypt `state/secrets/passwords/passwords.enc.env` to a secure temporary dotenv file
- encrypt a temporary dotenv file back to the encrypted password file atomically
- read dotenv values consistently, including quoted values
- quote dotenv values consistently before writing
- upsert key/value pairs consistently

## Updated consumers

The following areas now call the shared helper rather than carrying their own SOPS dotenv implementations:

- MikroTik common helpers
- Technitium credential bootstrap
- Technitium API token generation
- Technitium MikroTik DDNS render/deploy helpers
- Technitium Ansible playbook runner
- Proxmox Taskfile secret reads and upserts
- GitHub Taskfile password reads and upserts
- SSH Taskfile password loading
- normal password encrypt/decrypt/display tasks

## Intentional exceptions

`passwords:keys:rotate` and `passwords:audit` still use direct SOPS commands. They intentionally need special handling for alternate age keys and recovery diagnostics.

## Additional consistency fix

The `.sops.yaml` path rule generated by `passwords:keys:rotate` now matches the same encrypted password file path used by initial setup:

`^state/secrets/passwords/passwords.enc.env$`

Previously, key rotation widened the rule to all dotenv files under the password directory.

## Operational note

The original MikroTik failure was not caused by SOPS decryption. The log confirmed MikroTik credentials were present in the encrypted file before SSH failed. That failure still indicates stale or mismatched MikroTik credentials on the router.

Recommended recovery command sequence:

```bash
MIKROTIK_CREDENTIALS_FORCE=1 task mikrotik:credentials
task mikrotik:configure
task mikrotik:validate
```
